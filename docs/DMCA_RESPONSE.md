### `app/` audit — clean

- `app/main.py`: `logging.basicConfig` with default formatter. No structured exception-data capture. **No `sentry_sdk` import** — Sentry is not integrated in sift-api, so failure tracebacks are local to stdout only. CORS, security headers, rate limiting all clean.
- `app/db.py`: connection pool, migrations. No content handling.

### `Dockerfile` audit

Pure Python 3.12-slim. No Railway-specific bindings. Reads `${PORT:-8000}` env var. Portable to any container host (Fly.io, Render, Hetzner, GCP Cloud Run, AWS App Runner).

### `railway.toml` audit

Only deploy config (`healthcheckPath`, `restartPolicyType`, `restartPolicyMaxRetries`). Not used by other hosts but doesn't conflict.

### Summary: Sift's Railway-side DMCA exposure is **low**

| Concern | Risk |
|---|---|
| Original article body HTML stored on Railway disk | None |
| Article images downloaded + stored on Railway disk | None |
| Original article text in INFO/DEBUG log lines | None |
| Original article text in error tracebacks (Sentry/stdout) | None — Sentry not integrated; logger uses `%s` formatters without locals |
| Anthropic Batch API requests/responses persisted on Railway | None — requests are constructed transiently, JSONL results downloaded and processed in-memory |
| `content_hash` reveals body | None — SHA-256 of 500-char prefix is irreversible |

### 1.2 Minor flag: `summarizer.py` fallback truncation

When `_summarize_batch` raises (Claude API failure), the per-article fallback at `services/summarizer.py:42` stores `_truncate(article.raw_content, 200)` as the article's `summary` in Neon. That's up to 200 words of source content.

**Why it's low-risk:**

- The 200-word truncation is **substantially shorter** than the original (typical news articles are 800–2,000 words).
- The source for `raw_content` is the publicly published RSS feed, which the publisher voluntarily distributes for syndication.
- The truncation lives in **Neon Postgres**, not Railway disk — so it's outside Railway's fair-use clause scope.
- The fallback fires only on Claude API failure (rare).
- Cited with `source_url` and `source_name` (attribution preserved).

**Posture documented; no code change required.** If a publisher complains about this specific path, the fix is one line: `continue` in the except clause instead of storing truncated content. Defer until that complaint actually arrives — the engineering cost of "drop the article entirely on Claude failure" is occasional missing coverage in the feed, which the user notices more than the rare fallback summaries do.

## 2. Counter-notice template

If a DMCA takedown lands and the claim is **invalid** (the content is transformative, mis-identified, or fair use), file a counter-notice within 10 business days. Standard counter-notice form per 17 U.S.C. § 512(g)(3).

**Use only when the takedown is genuinely wrong. Don't bluff.** Bad-faith counter-notices have liability.

```
[YOUR NAME]
[YOUR ADDRESS]
[YOUR PHONE]
[YOUR EMAIL]

[DATE]

[SERVICE PROVIDER DMCA AGENT — e.g., Railway, dmca@railway.app]

RE: Counter-notification under 17 U.S.C. § 512(g)
    Reference: [TICKET / NOTICE ID]

To whom it may concern:

I am responding to your notice of [DATE] regarding the alleged
infringement of material located at [URL OF SIFT CONTENT, e.g.,
https://siftnews.kristenmartino.ai/article/{id}].

I have a good-faith belief that the material was removed or
disabled as a result of mistake or misidentification of the
material. Specifically:

  1. The Sift summary at the URL above is a TRANSFORMATIVE
     summary generated by an AI model (Claude Haiku 4.5),
     not a reproduction of the original article. The summary
     is 1–2 sentences, factually distinct from the original
     prose, and serves a different editorial purpose (a
     civic-literacy aid).

  2. Sift links back to the original source via a
     prominently displayed "Read at source" link. We do not
     republish article body content; we surface a short
     summary alongside attribution and a direct link.

  3. [If applicable: The article was discovered via the
     complainant's PUBLICLY PUBLISHED RSS FEED, which we
     consume under the implicit syndication license that
     RSS distribution constitutes.]

  4. [If applicable: The civic-context primer panel attached
     to this article is original editorial content generated
     by Sift's pipeline, not derived from the complainant's
     article. It defines third-party terms (politicians,
     orgs, bills) from public records (OpenSecrets, GovTrack,
     ProPublica, FARA) — none of which are the complainant's
     proprietary content.]

I consent to the jurisdiction of the U.S. District Court for
the [DISTRICT, typically Northern District of California for
Railway's principal place of business — verify when filing],
and I will accept service of process from the complainant or
the complainant's agent.

I swear, under penalty of perjury, that the above information
is accurate and that I have a good-faith belief that the
material was removed or disabled as a result of mistake or
misidentification.

Sincerely,

[SIGNATURE]
[YOUR NAME]
```

After filing, the service provider must restore content within 10–14 business days unless the complainant files suit. Document the entire exchange.

## 3. Response process

### When a notice arrives (first 24h)

1. **Forward the full notice** to your records — include sender, timestamps, claimed URLs, claimed copyrighted work.
2. **Identify the claimed content** — which article(s), which Sift surface (feed, dossier, primer, compare).
3. **Pull Railway logs** for the affected article IDs — confirm what was actually stored / served. Use `railway logs --service sift-api | grep $ARTICLE_ID`.
4. **Decide: comply, dispute, or partially comply.**

### Decision criteria

| Scenario | Action |
|---|---|
| Notice cites our 1–2 sentence AI summary on its own | **Dispute via counter-notice.** Transformative, attributed, links back. Standard fair use. |
| Notice cites a primer panel | **Dispute.** Primer is original editorial — defines third-party civic terms; doesn't reproduce article content. |
| Notice cites images | **Verify hot-linking.** All Sift images are hot-linked via RSS `image_url`. If we're hot-linking, we're not "hosting" — politely respond to that effect, no formal counter-notice needed. |
| Notice from a publisher whose RSS we ingest | **Reply explaining the syndication-license posture + offer to remove their outlet from the feed pool.** This is the cheapest peace. |
| Notice cites the rare `_truncate(raw_content, 200)` fallback path | **Comply** (remove the article + add the publisher to the "fallback-banned" list internally). The fallback isn't worth a fight. |
| Notice from AP, Reuters, NYT, NewsCorp, or another major publisher | **Get a 30-min lawyer consult before responding.** Stakes are high; templates aren't sufficient. EFF maintains a referral list. |
| Notice cites Sift's MCP-server compare workflow | **Dispute.** Compare uses Claude `web_search` — Anthropic's tool, not Sift's reproduction. |

### Time budget

- **Day 0**: receive notice. Acknowledge receipt to Railway / sender within 24h.
- **Day 1–3**: review and decide (see criteria above).
- **Day 3–5**: file counter-notice or comply.
- **Day 10**: hard deadline for counter-notice. After this, missing the deadline means the takedown stands.

### After resolution

- Document in `BACKLOG.md` under "Considered and rejected" if a complaint shapes future architecture.
- Update this runbook if the response pattern revealed gaps.

## 4. Host-portability check (Fly.io fallback)

If Railway pulls the service on a contested notice, sift-api can re-host on Fly.io in under an hour. Validated steps:

```bash
# 1. Install Fly CLI (if not already)
brew install flyctl  # macOS
# or: curl -L https://fly.io/install.sh | sh

# 2. From sift-api repo root
fly launch --no-deploy --name sift-api --region iad

# 3. Set env vars (mirror Railway's)
fly secrets set \
  ANTHROPIC_API_KEY=$ANTHROPIC_API_KEY \
  VOYAGE_API_KEY=$VOYAGE_API_KEY \
  DATABASE_URL=$NEON_URL \
  PIPELINE_API_KEY=$PIPELINE_API_KEY \
  ENVIRONMENT=production \
  LOG_LEVEL=info

# 4. Deploy
fly deploy
```

The existing `Dockerfile` works as-is — no Railway-specific bindings. The `railway.toml` is unused by Fly but doesn't conflict.

**Verification before declaring "ready":** smoke-test the deploy with `fly logs` + curl the health endpoint. Don't migrate prod DNS until you're confident.

**fly.toml template** (commit alongside Dockerfile when Fly.io becomes live):

```toml
app = "sift-api"
primary_region = "iad"

[build]
  dockerfile = "Dockerfile"
```
